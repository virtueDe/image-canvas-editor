# Figma 风格工作台 UI 重构设计文档

## 基本信息

- 日期：2026-03-29
- 主题：`apps/web-vue` 工作台 UI 重构
- 目标层：`apps/web-vue`
- 约束：不修改 `editor/core` 与 `editor/vue3` 的编辑行为与 API 语义

## Pre-Analysis Gate

### 1. 这是个真问题还是臆想出来的

这是真问题。当前页面采用常规文档流布局，头部、左侧工具区和画布区都在同一自然流里，导致两个直接后果：

1. 首屏可见画布面积不够大，编辑器不是第一视觉中心。
2. 展开功能组时，页面总高度继续增长，浏览器滚动条介入，破坏工作台体验。

### 2. 有更简单的方法吗

有。最小有效方案不是改编辑器内核，也不是加状态管理，而是把 `web-vue` 壳层改成固定工作台框架：

1. 固定头部承接全局动作。
2. 固定左侧检视器承接局部编辑工具。
3. 右侧画布工作区吃掉剩余空间。
4. 把滚动边界收口到左侧检视器内部。

### 3. 会破坏什么吗

如果越过 `web-vue` 边界，就有破坏上传、裁剪、草稿、导出和画布交互的风险。本次设计明确限制在页面壳与样式层，保持以下行为不变：

1. `useImageEditor()` 暴露的方法名和调用路径不变。
2. 上传、旋转、翻转、裁剪、滤镜、调节、文字、草稿、导出能力全部保留。
3. 画布交互语义不变，只调整其容器结构与展示方式。

## 用户已确认的约束

- 视觉方向：浅色、克制、接近 Figma 的专业工具台语言
- 桌面端布局：固定头部 + 固定左栏 + 画布优先
- 移动端策略：允许降级，左侧工具区改成抽屉或覆盖层
- 滚动策略：页面整体不滚动，左侧工具区内部滚动

## 目标与非目标

### 目标

1. 让画布成为首屏第一视觉中心，进入页面即可开始操作。
2. 把页面从“堆卡片的工具页”重构成“编辑器优先的固定工作台”。
3. 在保持全部现有功能的前提下，提升布局秩序、信息密度和操作效率。
4. 保证展开功能组时不会触发页面级滚动条。
5. 为移动端提供可用的降级方案，不为了强行常驻侧栏而压缩画布。

### 非目标

1. 不新增编辑功能，不变更任何编辑状态数据结构。
2. 不重写 `editor-core` 画布逻辑、历史记录逻辑和图像处理逻辑。
3. 不引入新的状态库、路由系统或设计系统框架。
4. 不把当前单页工作台拆成多路由应用。

## 现状问题

## Layer 1：数据结构分析

- 编辑数据和交互状态本身已经集中在 `editor-core` / `editor-vue`，边界是对的。
- 真问题不在数据结构，而在 `App.vue` 里把所有功能区和画布区放在同一套自然流布局中，导致“局部展开”被放大成“整页变高”。
- 当前 UI 壳没有明确定义“全局动作”“局部参数”“主要工作对象”三个层级。

## Layer 2：特殊情况识别

当前页面的几个特殊情况本质上都是布局结构错误带来的补丁：

1. 头部和画布都想抢首屏高度。
2. 左侧功能组展开后只能推高页面，而不是在自己的滚动容器内消化。
3. 空态提示、模式提示、缩放工具条都被迫和主体布局互相争夺空间。

这些都不该靠继续调间距或缩小字体解决，而应该通过固定工作台骨架把特殊情况消掉。

## Layer 3：复杂度审查

问题本质只有一句话：当前壳层把编辑工作流表达成了普通内容页，而不是固定工作台。

因此实现必须控制在两个文件层面为主：

1. `App.vue` 负责重排骨架与分区。
2. `styles.css` 负责视觉语言、滚动边界和响应式降级。

如果这次为了 UI 重构去新增复杂组件树或引入新依赖，就是过度设计。

## Layer 4：破坏性分析

潜在破坏点主要有四类：

1. 重排模板时遗漏原有事件绑定。
2. 固定布局后错误设置 `overflow`，导致画布或抽屉无法交互。
3. 移动端抽屉实现不当，导致页面锁滚逻辑和文件上传交互冲突。
4. 视觉改造过头，造成可读性下降或禁用态不清晰。

规避方式：

1. 保持所有编辑方法仍在 `App.vue` 调用，不改方法签名。
2. 只把滚动限制到明确容器，不对画布交互层做侵入式包裹。
3. 移动端仅做展示层降级，不改业务条件和状态切换逻辑。
4. 先完成结构重排，再统一视觉变量，最后补移动端抽屉。

## Layer 5：实用性验证

这是生产中真实会影响体验的问题：

1. 首屏画布太小直接影响核心任务效率。
2. 页面级滚动条让“固定工作台”预期被打破。
3. 过多说明文案和卡片间距挤占了本就有限的可视空间。

解决方案的复杂度和问题严重性是匹配的，因为它只动壳层，不碰核心逻辑。

## 推荐方案

采用“浅色 Figma 风固定工作台”方案。

### 方案概要

1. 顶部使用固定全局工具条，承接上传、草稿、导出等高频全局动作。
2. 左侧使用固定检视器，宽度收敛到约 `280px`，内容区独立滚动。
3. 右侧使用固定主工作区，头部为轻量状态条，下方画布区占满剩余空间。
4. 页面整体锁定为 `100vh` 工作台，不依赖文档流撑高页面。
5. 移动端改为“顶部栏 + 工具抽屉 + 画布优先”的降级模式。

### 为什么推荐它

1. 它是最小代价解决真问题的方案，且与用户需求完全一致。
2. 改动集中在 `apps/web-vue`，兼容风险可控，回滚简单。
3. 它消除了“展开面板导致整页滚动”的结构性问题，而不是继续打补丁。

## 方案对比

### 方案 A：固定头部 + 固定左栏 + 右侧画布自适应

优点：

1. 最符合 Figma 类专业工具台体验。
2. 画布空间最大，页面层级最清晰。
3. 可以彻底避免展开工具组时出现页面滚动条。

缺点：

1. 需要补一套移动端抽屉降级。
2. `App.vue` 结构会比当前版本更明确、更刚性。

### 方案 B：保留当前自然流，只压缩间距和卡片高度

优点：

1. 改动最小。
2. 视觉风险较低。

缺点：

1. 无法根治页面滚动问题。
2. 画布空间提升有限。
3. 本质上仍然不像工作台，只是更紧的表单页。

### 方案 C：改成图标导航 + 浮动属性面板

优点：

1. 画布能更大。
2. 风格更像重型设计工具。

缺点：

1. 改动过大，超出本次“最小改动重构 UI”的范围。
2. 学习成本上升，信息架构变化太激进。

结论：采用方案 A。

## 信息架构

### 顶部固定栏

职责：

1. 承接品牌标题与极简副标题。
2. 承接全局动作：`上传图片`、`保存草稿`、`恢复草稿`、`导出 PNG`。

规则：

1. 不做营销 Hero。
2. 不放长说明，不抢画布注意力。
3. 固定在顶部，始终可达。

### 左侧固定检视器

职责：

1. 承接局部编辑工具。
2. 提供分组折叠和模式高亮。
3. 通过内部滚动吸收展开内容，不影响页面整体高度。

建议分组顺序：

1. 图片信息
2. 旋转与翻转
3. 裁剪
4. 文字
5. 滤镜预设
6. 参数调节

规则：

1. `图片信息`、`旋转与翻转`、`文字`、`参数调节` 默认展开。
2. 进入裁剪模式时，`裁剪` 分组自动展开并高亮。
3. 每组说明文案减量，只保留真正帮助操作的提示。

### 右侧主工作区

职责：

1. 顶部显示模式状态、撤销重做、缩放控制、复位视图。
2. 中央画布容器占满剩余高度，成为主舞台。
3. 空态提示和模式提示附着在画布区，不再占据独立垂直空间。

规则：

1. 画布区必须在桌面首屏可见且可操作。
2. 工作区高度随视口变化，不依赖页面滚动。
3. 模式提示浮层轻量化，只提供当前操作最关键的信息。

## 视觉语言

### 总体方向

采用浅色专业工具台语言，参考 Figma 的信息组织与表面秩序，但不机械复刻：

1. 整体改为浅灰白基底。
2. 面板、工具条和画布区用清晰但克制的分层。
3. 圆角、边框、阴影全部收敛，强调“精确”和“可操作”。

### 色彩

建议基调：

1. 页面背景：冷静的浅灰工作区色。
2. 面板背景：白色或极浅灰。
3. 边框：低对比细描边。
4. 强调色：单一蓝色系或中性偏蓝强调色，用于当前激活、主按钮、聚焦态。

### 字体与密度

1. 维持高可读中文无衬线字体栈。
2. 标题缩短、说明减量、参数层级明确。
3. 按钮和分组标题密度提高，但不压到误触。

### 组件观感

1. 面板像工具，而不是像装饰卡片。
2. 滑杆、按钮、折叠头部统一成一套轻量器材感控件语言。
3. 画布容器视觉上是绝对主舞台，四周辅助控件退居边缘。

## 交互设计

### 页面滚动策略

桌面端：

1. `body` / 根容器锁定为工作台高度。
2. 页面整体不滚动。
3. 左侧检视器内容区允许内部滚动。
4. 右侧主工作区不出现无意义的外层滚动条。

移动端：

1. 页面继续控制为单屏优先。
2. 左侧检视器改为抽屉或覆盖层。
3. 抽屉打开时锁定背景滚动。

### 空态

目标：

1. 让用户进入页面就知道这里是编辑工作台。
2. 指引足够直接，但不啰嗦。

内容：

1. 主文案聚焦“上传图片开始编辑”。
2. 辅文案简述裁剪、旋转、调色、导出。
3. 不使用营销型大段介绍。

### 裁剪模式

1. 左栏 `裁剪` 分组高亮并自动展开。
2. 右侧模式标签明确显示当前处于裁剪模式。
3. 保持现有裁剪交互，不改拖拽、缩放和应用逻辑。

### 文字模式

1. 左栏 `文字` 分组在出现文字叠加层时保持可见。
2. 模式提示区明确说明可拖动画布中文字定位。
3. 不改文字叠加的撤销、重做和导出语义。

### 缩放与视图控制

1. 缩放、撤销、重做、复位视图放在右侧工作区顶部工具条。
2. 信息更紧凑，减少高度占用。
3. 这些控件始终首屏可达。

## 组件与结构调整建议

### `App.vue`

负责：

1. 顶层固定工作台框架。
2. 桌面与移动端结构切换。
3. 检视器分组状态与当前模式高亮。
4. 维持 `useImageEditor()` 的现有调用关系。

### `WorkbenchHeader.vue`

继续保留，但职责进一步收敛：

1. 只负责固定头部内容。
2. 只承接全局动作，不承接局部编辑信息。

### `InspectorSection.vue`

继续保留，但需要服务固定检视器：

1. 提供统一折叠头。
2. 支持紧凑模式和高亮态。
3. 内容区域不影响外层布局高度。

### `styles.css`

负责：

1. 全局浅色变量。
2. 固定布局、滚动边界和响应式降级。
3. 检视器、顶部栏、主工作区、画布舞台的统一视觉。

## 实现边界

允许改动：

1. `apps/web-vue/src/App.vue`
2. `apps/web-vue/src/styles.css`
3. `apps/web-vue/src/components/WorkbenchHeader.vue`
4. `apps/web-vue/src/components/InspectorSection.vue`
5. 如有必要，可补少量纯展示组件

禁止改动：

1. `editor/core` 的状态、渲染和图像处理逻辑
2. `editor/vue3` 的桥接方法签名和语义
3. 任何会改变草稿结构、导出结果或裁剪坐标语义的代码

## 实施顺序

### Patch 1：重排固定工作台骨架

1. 把页面外层改成固定头部 + 左栏 + 主工作区。
2. 把页面滚动边界收口到左栏内部。
3. 保证右侧画布区在桌面首屏最大化可见。

### Patch 2：统一浅色视觉系统

1. 重写颜色变量、边框、阴影、面板层级。
2. 收敛说明文字与卡片密度。
3. 让工具控件与画布舞台形成清晰主次。

### Patch 3：补移动端抽屉降级

1. 引入移动端工具抽屉。
2. 抽屉打开时锁定背景滚动。
3. 确保手机上优先看到画布，而不是先看到一整列工具。

### Patch 4：状态与细节打磨

1. 模式标签、空态、提示浮层统一风格。
2. 检查折叠分组与裁剪/文字模式联动。
3. 清理旧类名和漏网视觉残留。

## 验证方案

### 自动验证

1. 运行 `pnpm --filter @image-canvas-editor/web-vue typecheck`

### 手工验证

桌面端：

1. 页面加载后无页面级纵向滚动条。
2. 顶部栏固定，左侧栏固定，右侧画布首屏可操作。
3. 展开多个功能组后，只有左栏内容区滚动。
4. 上传图片后画布显示正常。
5. 旋转、翻转、裁剪、文字、滤镜、调节、撤销重做、缩放、复位都可用。
6. 保存草稿、恢复草稿、导出 PNG 正常。

移动端：

1. 首屏优先显示画布。
2. 工具栏以抽屉或覆盖层方式打开。
3. 打开抽屉时背景不滚动，关闭后恢复。
4. 主要操作仍可完成，不出现明显布局炸裂。

## 风险与回滚

### 风险

1. 固定布局处理不当，可能让某些浏览器出现双滚动条。
2. 移动端抽屉与视口高度计算可能出现遮挡。
3. 模板重排时可能漏掉原有事件或禁用态绑定。

### 回滚策略

1. 改动严格限制在 `web-vue`。
2. 以小 patch 推进，结构、视觉、移动端降级分开提交。
3. 一旦出现交互回归，优先回退对应 UI patch，不触碰核心编辑器代码。

## 交付定义

当满足以下条件时，本次 UI 重构视为完成：

1. 桌面端呈现出清晰的固定工作台结构。
2. 顶部和左侧固定，画布面积显著大于当前版本。
3. 展开功能组时不再出现页面级滚动条。
4. 浅色 Figma 风格成立，但不影响控件可读性和可操作性。
5. 移动端提供抽屉式降级并保持基本可用。
6. 现有编辑能力全部保持兼容并通过验证。
