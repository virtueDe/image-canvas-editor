# 多文字 Canvas 原位编辑设计

## 摘要

本设计将当前图片编辑器的单文字覆盖层升级为多文字对象系统，支持以下能力：

- 一次添加多处文字
- 点击工具栏“添加文字”后进入插入态，再在画布点击落点创建文字
- 点击已有文字时直接进入原位编辑
- 编辑态在画布中显示编辑框、自定义插入光标与拖拽图标
- 按回车插入换行
- 右侧属性面板只编辑当前选中的文字对象

本次方案采用 `Canvas 可见渲染 + 隐藏 textarea 输入代理` 的路线。也就是：所有用户可见的文字、选中框、插入光标、拖拽手柄都由 Canvas 绘制；浏览器中的隐藏 `textarea` 只负责接收键盘输入、IME、粘贴和删除操作。

这是一个真实问题，不是过度设计。当前 `editor/core` 只有一个 `textOverlay`，无法表达多文字对象、当前选中项、插入态和编辑态。继续在旧结构上打补丁，只会把特殊情况堆成垃圾。开发期还没有正式用户，因此不做旧草稿兼容，直接切换到新草稿结构。

## 背景与目标

### 现状

当前实现有以下结构性限制：

- `EditorState` 中只有一个 `textOverlay`
- 命中检测、渲染和导出都只考虑单段单行文字
- 右侧面板假定只有一段文字可编辑
- 画布交互只支持拖动唯一文字对象

这些限制与目标需求直接冲突。用户现在要的是多段文字、点击已有文字立即编辑、画布原位输入、回车换行，以及右侧属性面板始终跟随当前文字对象。

### 本次目标

1. 将单文字模型升级为多文字对象模型。
2. 支持插入态下在画布任意位置创建新文字。
3. 支持点击已有文字立即进入原位编辑。
4. 支持编辑框、拖拽手柄、自定义鼠标指针与自绘插入光标。
5. 支持显式换行。
6. 支持右侧属性面板只编辑当前选中项。
7. 保持撤销重做、PNG 导出、裁剪模式锁定等现有边界不被破坏。

### 非目标

- 不实现富文本样式混排
- 不实现自动换行宽度约束
- 不实现文字图层列表
- 不兼容旧草稿结构
- 不实现字级精确鼠标落点定位的完整排版系统

## 方案选择

比较过三种路线后，最终采用下述方案：

### 采用方案

`Canvas 可见渲染 + 隐藏 textarea 输入代理`

- Canvas 是唯一可见文本层
- 文本对象、选中框、拖拽手柄、插入光标都由渲染器统一绘制
- 隐藏 `textarea` 仅接收键盘输入和 IME 事件
- `editor/core` 维护文本对象和文本工具状态机
- `editor/vue3` 负责把 DOM 输入事件桥接到 core
- `apps/web-vue` 负责焦点管理、指针样式和右侧属性面板

### 不采用方案

#### 方案 1：只在编辑态显示 DOM 文本框

这个方案复杂度更低，但用户明确要求采用方案 2 的体验，也就是画布统一绘制编辑态。

#### 方案 3：所有文字长期保持为 DOM 图层

该方案会把渲染、命中检测、导出与历史管理拆成两套世界，和当前仓库边界冲突太大，不适合现阶段。

## 架构设计

### 分层职责

#### `editor/core`

负责：

- 文本对象数据模型
- 文本工具状态机
- 命中检测
- 文本布局
- 预览渲染
- 导出渲染
- 撤销重做快照边界
- 草稿序列化结构

不负责：

- 真正的 DOM 输入控件
- `window.alert`
- 文件选择器
- 浏览器原生焦点事件细节

#### `editor/vue3`

负责：

- 将 core 暴露为 Vue 可消费状态
- 暴露当前选中文字、编辑态、插入态等计算状态
- 桥接隐藏 `textarea` 的输入、粘贴、组合输入、键盘事件

不负责：

- 保存第二份文本草稿
- 独立维护文本编辑器状态

#### `apps/web-vue`

负责：

- 画布外的隐藏 `textarea`
- 插入态与编辑态的焦点切换
- 工具栏“添加文字”按钮
- 右侧属性面板
- 画布上拖拽图标对应的指针反馈

不负责：

- 文本内容真相
- 文本排版算法
- 导出绘制逻辑

## 数据结构

### 新的数据模型

建议将当前 `textOverlay` 替换为以下结构：

```ts
export interface TextItem {
  id: string;
  content: string;
  xRatio: number;
  yRatio: number;
  fontSize: number;
  color: string;
  align: 'left' | 'center' | 'right';
  lineHeight: number;
}

export type TextToolState =
  | { mode: 'idle'; hoverTextId: string | null }
  | { mode: 'inserting' }
  | {
      mode: 'editing';
      textId: string;
      caretIndex: number;
      selectionStart: number;
      selectionEnd: number;
      composing: boolean;
    }
  | {
      mode: 'dragging';
      textId: string;
      startClientX: number;
      startClientY: number;
      originXRatio: number;
      originYRatio: number;
    };
```

`EditorState` 中的相关字段调整为：

```ts
interface EditorState {
  ...
  texts: TextItem[];
  activeTextId: string | null;
  textToolState: TextToolState;
  ...
}
```

### 设计原则

- 文本对象是长期数据，必须可序列化、可导出、可进入历史
- `hoverTextId`、插入光标闪烁、组合输入中的临时状态属于瞬时 UI 状态，不进入历史
- 当前选中项由 `activeTextId` 唯一标识，右侧面板和渲染都只认这一份真相

## 状态机

### 状态定义

#### `idle`

- 没有插入操作
- 可以 hover 文字对象
- 点击已有文字可直接进入 `editing`

#### `inserting`

- 点击“添加文字”后进入
- 鼠标指针切换为插入态样式
- 在画布点击任一点后创建新文字，并立即进入 `editing`

#### `editing`

- 当前文字显示编辑框
- 编辑框固定带一个拖拽图标
- 画布中显示自绘插入光标
- 键盘输入全部写入当前文字对象

#### `dragging`

- 只允许从编辑框上的拖拽图标进入
- 拖动完成后回到 `editing`
- 当前文字仍保持选中

### 状态切换规则

1. 点击“添加文字” -> `idle` 到 `inserting`
2. 插入态点击画布 -> 创建文字 -> `editing`
3. 点击已有文字 -> `editing`
4. 按住拖拽图标并移动 -> `dragging`
5. 松开拖拽 -> `editing`
6. 点击空白区域 -> 结束编辑，回到 `idle`
7. 若结束编辑时文字内容为空 -> 删除该对象，再回到 `idle`

### 为什么这样设计

关键点是消灭“点击文字既可能是编辑又可能是拖动”的歧义。正文区域永远负责编辑，拖拽图标永远负责移动。这样状态机干净，不需要靠热区猜测用户意图。

## 输入模型

### 隐藏 textarea 的职责

隐藏 `textarea` 只负责：

- 接收普通输入
- 接收回车并转为 `\n`
- 接收退格、删除、方向键等编辑指令
- 接收 IME 组合输入
- 接收粘贴

它不负责：

- 可见排版
- 可见选区
- 可见光标

### 推荐的输入接口

`editor/core` 暴露明确的编辑 API，例如：

- `startTextInsertion()`
- `placeTextAt(xRatio, yRatio)`
- `focusText(textId)`
- `insertText(text: string)`
- `insertLineBreak()`
- `deleteBackward()`
- `deleteForward()`
- `setSelection(start: number, end: number)`
- `finishTextEditing()`
- `startDraggingText(textId, ...)`
- `updateDraggingText(...)`
- `finishDraggingText()`

这样可以避免把 DOM 事件细节泄漏到 core。

## 文本布局与渲染

### 布局规则

- 文本内容原样保存为字符串，换行使用 `\n`
- 当前阶段只支持显式换行，不做自动换行
- 多行布局时，按行拆分内容，逐行测量宽度
- 文本整体包围盒按“最大行宽 + 总行高”计算
- 文本对齐按对象属性决定

### 统一布局模块

当前 `text-overlay.ts` 只服务单段单行文字。新方案应将其升级为通用文本布局模块，由以下两个调用方共用：

- `renderer.ts`：画布预览
- `image-processing.ts`：PNG 导出

这是为了避免预览和导出使用两套算法导致位置漂移。

### 编辑态的画布绘制内容

编辑态至少包含以下可见元素：

- 当前文字内容
- 编辑框边界
- 拖拽图标
- 自绘闪烁插入光标
- 可选的选区高亮

其中拖拽图标建议固定在编辑框右上角或左上角，并为其提供比视觉图标更大的命中区域。

## 交互设计

### 插入新文字

1. 用户点击“添加文字”
2. 进入插入态
3. 用户在画布点击落点
4. 创建新文字对象并设为当前选中项
5. 自动聚焦隐藏 `textarea`
6. 画布显示编辑框和插入光标

### 编辑已有文字

1. 用户点击已有文字
2. 该文字成为 `activeTextId`
3. 直接进入编辑态
4. 画布显示编辑框和拖拽图标
5. 隐藏 `textarea` 聚焦

### 拖动文字

1. 用户按住编辑框上的拖拽图标
2. 进入 `dragging`
3. 根据指针移动更新文字的 `xRatio` 与 `yRatio`
4. 释放指针后回到 `editing`

### 结束编辑

- 点击画布空白处：结束编辑
- 若文本为空：自动删除该对象
- 否则保留对象并回到 `idle`

## 右侧属性面板

### 面板原则

- 只编辑当前选中对象
- 不显示文字列表
- 不保存第二份草稿

### 面板内容

当没有选中文字时：

- 显示空态提示，例如“点击画布中的文字开始编辑”
- 内容、字号、颜色、对齐、行高控件禁用

当有选中文字时：

- 文本内容
- 字号
- 颜色
- 对齐方式
- 行高
- 删除当前文字

### 文本内容入口策略

画布原位编辑是主入口。右侧内容输入框仍可保留，但它只是当前对象的辅助编辑入口，修改后应同步回写同一 `TextItem`，不能形成第二套状态。

## 撤销重做策略

### 进入历史的内容

必须进入历史：

- 新建文字
- 删除文字
- 拖拽结束后的最终位置
- 右侧属性修改完成后的最终状态
- 一次文本编辑会话完成后的文本结果

不进入历史：

- hover 状态
- 插入光标闪烁
- 当前是否处于组合输入中
- 正在拖拽过程中的中间帧

### 文本编辑历史边界

文本输入不能按单字符入栈，否则撤销体验会劣化。推荐策略：

- 进入 `editing` 时记录基线快照
- 在编辑过程中只更新当前状态，不立即提交历史
- 结束编辑时将整个编辑会话作为一条历史记录提交

## 草稿策略

本次不做旧草稿兼容。

原因：

- 项目仍处于开发期
- 当前只有开发者本人测试
- 为旧结构写迁移层只会污染模型演进

建议做法：

- 切换到新的草稿 key，例如 `image-canvas-editor:draft:v2`
- 或保留统一 key，但提升 `schemaVersion`
- 恢复失败或结构不匹配时，直接视为“无可恢复草稿”

## 裁剪、视图与导出边界

### 裁剪模式

- 保持现有规则：裁剪模式下禁用文字编辑
- 进入裁剪模式时若处于文字编辑中，应先结束或取消编辑态

### 视图缩放与平移

- 文字对象的存储坐标仍使用相对坐标比值
- 预览中的缩放和平移只影响显示，不直接写坏文字对象
- 文本命中检测、编辑框、拖拽图标都应基于统一的屏幕映射结果

### 导出

导出只绘制最终文字内容，不绘制：

- 编辑框
- 拖拽图标
- 自绘插入光标
- 选区高亮

## 测试策略

### `editor/core` 单元测试

应新增或重构以下测试：

- 多文字对象创建与删除
- 插入态点击后创建文字并激活
- 点击已有文字进入编辑态
- 拖拽图标移动后的坐标更新
- 多行文本包围盒计算
- 显式换行的导出布局
- 文本编辑会话只生成一条历史记录
- 空文字结束编辑时自动删除

### `editor/vue3` 桥接测试

- 当前选中文字是否正确暴露
- 文本输入桥接是否正确调用 core
- 插入态和编辑态的焦点切换是否稳定

### `apps/web-vue` 交互验证

- 点击“添加文字”进入插入态
- 在画布点击后开始编辑新文字
- 点击已有文字显示编辑框
- 拖拽图标可移动文字
- 右侧面板跟随当前选中对象变化

## 实施顺序建议

1. 重构 `types.ts` 与 `EditorState`，引入 `TextItem[]`、`activeTextId`、`textToolState`
2. 提取并重写文本布局模块，使其支持多行文本
3. 改造 `renderer.ts` 和 `image-processing.ts`，共用统一布局
4. 改造 `editor.ts` 的文本交互状态机
5. 在 `editor/vue3` 增加隐藏 `textarea` 桥接接口
6. 重构 `apps/web-vue` 的右侧文字面板和画布编辑态 UI
7. 补齐 `editor/core` 为主的测试

## 风险与缓解

### 风险 1：文本布局与导出不一致

缓解：

- 预览和导出共用同一布局函数
- 为多行文本导出单独补测试

### 风险 2：编辑态和拖拽态冲突

缓解：

- 正文区域只负责编辑
- 只有拖拽图标负责移动

### 风险 3：IME 与隐藏 textarea 不稳定

缓解：

- DOM 输入事件只做桥接，不在 UI 壳层保存文本真相
- 优先处理 `compositionstart/update/end`、`beforeinput`、`input`

### 风险 4：一次性改动面过大

缓解：

- 先改 core 数据模型和渲染
- 再接入 Vue 桥接和右侧面板
- 每一步都以可逆 patch 推进

## 最终结论

这次改动的关键不是“把输入框挪到画布上”，而是把文本系统从“单对象补丁”升级为“多对象 + 明确状态机”的设计。只要数据结构和状态边界正确，原位编辑、拖拽图标、右侧属性面板、撤销重做和导出都能自然落地。反过来，如果还试图沿用单一 `textOverlay`，后续每加一个需求都会继续制造特殊情况。
