# UI 改造设计文档

## 基本信息

- 日期：2026-03-26
- 主题：当前项目界面 UI 改造
- 目标层：`apps/web-vue`
- 约束：不修改 `editor/core` 与 `editor/vue3` 的现有编辑行为

## Pre-Analysis Gate

### 1. 这是个真问题还是臆想出来的

这是真问题。当前 [App.vue](C:/code/virtue/image-canvas-editor/apps/web-vue/src/App.vue) 功能完整，但页面表达停留在“可用工具页”，没有形成成熟创作者工具应有的层级、质感与工作台体验。

### 2. 有更简单的方法吗

有。最小有效方案不是重写编辑器，也不是新增状态层，而是只在 `web-vue` 壳层内重构页面骨架、视觉语言与信息层级。

### 3. 会破坏什么吗

如果改动越过 `apps/web-vue` 边界，就有破坏裁剪、草稿、导出和编辑状态的风险。本次方案明确限制在 UI 壳层，保持现有功能入口与调用链兼容。

## 目标与非目标

### 目标

1. 把当前页面改造成“编辑器优先”的创作者工作台，而不是普通表单式工具页。
2. 在不引入品牌叙事负担的前提下，建立稳定、克制、专业的产品气质。
3. 提升桌面端的视觉层级和操作秩序，让画布成为第一视觉中心。
4. 保留全部现有编辑能力与调用路径，避免行为回归。

### 非目标

1. 不新增编辑能力，不改动核心数据结构。
2. 不重写 `editor-core` 的画布交互、导出、草稿与裁剪逻辑。
3. 不把页面改造成营销型产品首页。
4. 不以移动端为优先目标，移动端只要求可用、不炸布局。

## 用户已确认的设计约束

- 产品定位：创作者工具
- 首屏策略：编辑器优先
- 视觉方向：工作室编辑风
- 设备优先级：桌面优先
- 面板组织：分组折叠
- 品牌表达：弱品牌

## 现状问题

### 数据与边界

- `web-vue` 目前同时承担页面布局、控件摆放和视觉表达，这是合理边界。
- 真正的问题不在状态数据，而在 UI 结构把所有控制面板并列堆叠，导致功能虽然齐全，但主次不清。

### 特殊情况

- 当前页面使用统一卡片堆叠处理所有工具块，缺少“高频动作”和“低频设置”的区分。
- 画布区虽然居中，但上层浮动提示与左侧控制区的视觉竞争明显，作品本身不是第一视角。

### 复杂度

- 功能项不算多，问题本质是一句话：现有页面把创作工作流表达成了后台配置页。
- 解决方案应优先通过布局与视觉层级重塑，而不是增加新的交互系统。

### 兼容性

- 任何导致上传、旋转、翻转、裁剪、预设、调节、草稿、导出入口消失或行为变化的改动都视为回归。

## 推荐方案

采用“工作室工作台”方案：

- 保持页面为单页编辑器，不拆成营销页与编辑页两套路由。
- 顶部使用轻量头部承接产品名、简短说明和高频动作。
- 左侧工具区改造成分组折叠的检视器。
- 中央区域改造成具有作品台面感的大画布舞台。
- 空态、裁剪态、缩放区和提示区全部统一为同一套视觉语言。

选择该方案的理由：

1. 能解决当前主次失衡的问题。
2. 改动集中在 `apps/web-vue`，兼容风险低。
3. 不需要新增全局状态，也不需要侵入核心编辑器逻辑。

## 信息架构

### 顶部轻量头部

职责：

- 承接通用产品名和一句简短说明。
- 放置高频全局动作。

保留动作：

1. `选择图片`
2. `保存草稿`
3. `恢复草稿`
4. `下载 PNG`

原则：

- 顶部不是 Hero，不承担完整营销职责。
- 头部信息密度低于工具区，避免与画布争夺注意力。

### 左侧工具检视器

分组顺序：

1. 图片信息
2. 旋转与翻转
3. 裁剪
4. 滤镜预设
5. 参数调节

交互规则：

- 使用折叠分组减少首屏噪声。
- 默认展开 `图片信息`、`旋转与翻转`、`参数调节`。
- `裁剪` 分组在进入裁剪模式时自动高亮，必要时自动展开。

### 中央画布舞台

职责：

- 成为页面视觉重心。
- 保留当前缩放与视图复位能力。
- 承接空态提示、缩放控件、模式提示。

布局原则：

- 画布外层需要更明显的留白与边界控制。
- 控件附着在画布边缘，不抢主视觉中心。
- 作品板和工作台背景形成层级对比。

## 视觉语言

### 核心方向

采用“工作室编辑风”，强调克制、质感和作品优先：

- 背景使用暖中性色工作台体系，而不是冷色霓虹科技感。
- 强调材质、留白、细边框和柔和阴影，而不是大面积发光。
- 品牌感来自整体秩序与表面质感，而不是营销语或复杂叙事。

### 颜色

建议主色系：

- 基底色：深墨灰、烟棕、纸白
- 强调色：单一高识别强调色，用于主按钮、激活态和关键数值
- 语义色：仅在必要状态中少量使用

### 字体

- 标题采用更具编辑气质的标题字体，建立识别度。
- 正文与参数区使用高可读中文无衬线字体。
- 不继续依赖默认 `Inter/system` 风格作为唯一表达。

### 控件

- 面板采用浅暖半透明表面，配细边框和轻阴影。
- 滑杆、折叠标题、预设按钮统一成“器材感”控件语言。
- 预设区视觉上更像风格样片选择，但不依赖真实缩略图素材。

## 关键交互

### 空态

未上传图片时，画布区显示专业但克制的工作台空态：

- 主引导为导入图片开始编辑。
- 辅助说明列出核心能力，如裁剪、旋转、预设、导出。
- 不出现营销型冗长文案。

### 裁剪模式

- 进入裁剪后，左侧对应分组高亮。
- 画布区提示文案同步切换，明确当前处于裁剪模式。
- 保留当前裁剪交互本身，不改核心行为。

### 预设与参数调节

- 预设区从普通按钮升级为更有风格识别的选择项。
- 参数调节继续使用滑杆，但强化当前值、激活态和读数层级。

### 缩放与视图控制

- 控件继续保留在画布区。
- 位置更贴边，视觉重量降低。
- 信息优先级低于画布内容，但高于背景提示。

## 实现边界

允许改动：

- `apps/web-vue/src/App.vue`
- `apps/web-vue/src/styles.css`
- `apps/web-vue/uno.config.ts`
- 如有必要，可新增 `apps/web-vue/src/components/` 下的纯展示组件

禁止改动：

- `editor/core` 中的编辑器状态与图像处理逻辑
- `editor/vue3` 中的桥接 API 语义
- 现有导入、导出、草稿恢复的行为契约

## 实施建议

建议拆成小 patch：

1. 先重排骨架与信息架构，不大改视觉细节。
2. 再统一视觉变量、面板、按钮和滑杆样式。
3. 最后补空态、裁剪态提示和桌面端细节打磨。

这样做的原因：

- 便于逐步验证绑定关系与禁用态。
- 出现回归时更容易回滚。
- 可以避免“结构与皮肤一起爆炸”的排查成本。

## 验证方案

### 必做检查

1. 上传图片
2. 左右旋转
3. 水平与垂直翻转
4. 进入裁剪、应用裁剪、取消裁剪、清除裁剪
5. 切换滤镜预设
6. 调整对比度、曝光、高光
7. 保存草稿
8. 恢复草稿
9. 导出 PNG

### 额外人工检查

1. 桌面端主视口层级是否清晰
2. 空态是否自然
3. 折叠面板是否稳定
4. 裁剪模式提示是否明确
5. 缩放控件是否不抢主视觉
6. 窄屏下是否至少可用且无明显布局破坏

## 风险与回滚

### 风险

1. 结构重排可能导致按钮绑定遗漏。
2. 折叠状态可能和裁剪模式提示不同步。
3. 视觉重做可能影响现有禁用态、可读性和焦点层级。

### 回滚策略

- 所有改动保持在 `web-vue`。
- 如果出现问题，优先按 patch 回退页面壳改动。
- 不触碰核心编辑器逻辑，保证草稿与导出兼容性不受影响。

## 交付定义

当满足以下条件时，本次 UI 改造视为完成：

1. 页面呈现出明确的“编辑器优先”工作台结构。
2. 视觉风格符合“工作室编辑风”且品牌表达保持克制。
3. 左侧工具区实现分组折叠并维持合理默认展开。
4. 画布成为页面第一视觉中心。
5. 全部现有编辑能力保持可用并通过手工验证。
